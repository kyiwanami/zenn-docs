## はじめに

Amplify + Figma を中心に開発していると、次のような壁にぶつかりやすい。

- Figma運用の複雑さ
- デザインの属人性
- 反映時間の長さ。特に `amplify pull` を含む更新確認はテンポを落としやすい
- Amplify Gen2 との互換性をどう担保するか

こうした悩みは、以前の取り組みでも触れてきたが、実務では「デザインを維持しながら、もっと早く安全に画面を作れないか」という課題に行き着くことが多かった。参考として、以前の記事導線は[こちら](https://trust-coms.com/)。

そこで有効だったのが、MUI の豊富なコンポーネント群と Cursor の組み合わせである。Figma で得ていた見た目の意図を保ちながら、実装速度と保守性を引き上げる方向に舵を切った。

## 補足: Cursorエディタについて

Cursor はエディタに統合された AI 支援環境として使えるため、実装中のファイルや周辺コードを見ながらそのまま対話できるのが強い。特に便利なのが `@` によるコンテキスト参照で、関連ファイルや要件定義をまとめて指示に含められる。

単に「画面を作って」と依頼するのではなく、既存コンポーネント、型定義、親子関係、業務要件を一緒に渡せるので、生成コードの精度が上がりやすい。

## 具体的な移行の3ステップ

### ステップ1 準備フェーズ

最重要なのは、UI 実装そのものより先に業務理解を固めることだ。理由は明確で、以下の3点に直結するからである。

- 適切なコンポーネント設計ができる
- Cursor への指示が具体的かつ正確になる
- 生成コードが業務要件を満たしているか評価できる

実施例としては、まず業務関連ドキュメントを Markdown 化して集約する。仕様書、画面説明、入力ルール、エラー条件などを検索しやすい形でまとめておくと、Cursor に渡す文脈としても使いやすい。情報が不足している場合は、必要に応じてヒアリングも行う。

### ステップ2 実装フェーズ

#### ① 段階的な置き換え戦略

最初から全画面を置き換えるのではなく、一部のコンポーネントや画面単位で段階的に MUI 化する。これにより、既存の Figma ベース実装を残したまま比較できるため、リスクを抑えて移行できる。

#### ② 並行開発を可能にする設計

既存ファイルを直接壊すのではなく、別ファイルとして新実装を作る。たとえば `src/components/UserInputForm.tsx` が既存実装なら、MUI 版として `UserInputFormMui.tsx` を追加する。

切り替えには React Context を使い、開発環境では比較用スイッチを置いておくと便利だ。既存版と MUI 版を並行して確認できるので、UI の差分や入力体験の差をすぐ見比べられる。

#### ③ 効果的なCursorへの指示方法

Cursor には、関連ファイルを `@` で含めて指示するのが重要だ。最低でも次の情報は一緒に渡したい。

- 元コンポーネント
- 型定義
- 親子関係にあるコンポーネント
- 業務要件定義ファイル

具体的には、以下のような形で依頼できる。

```text
@UserInputForm.tsx (Figmaベース) を参考に、MUIコンポーネントを使用して UserInputFormMui.tsx を作成してください。
参考情報
- データ型: @src/types/user.ts
- 親コンポーネント: @UserRegistrationPage.tsx
実装要件
- 以下のユーザー情報を登録・編集するフォームを作成してください
  - 名前 (必須、最大50文字)
  - メールアドレス (必須、メール形式チェック)
  - 年齢 (任意、数値のみ)
- 登録ボタンは、全ての必須項目が入力され、バリデーションエラーがない場合のみ活性化してください。
- 編集モードの場合、初期値がフォームに表示されるようにしてください。
```

このように要件を具体化しておくと、見た目だけではなくバリデーションや編集モードまで含めた実装に近づきやすい。

#### ④ AI生成コードのレビューは必須

Cursor がコードを生成してくれても、レビューは必須である。特に業務ロジック、入力制約、例外系の扱いは、人間が確認しないと危険だ。

また、Cursor には Rules for AI の考え方もあり、公式ドキュメントは[Cursor Docs](https://docs.cursor.com/)で確認できる。ルール整備とレビュー工程を組み合わせることで、生成品質のブレを抑えられる。

#### ⑤ TypeScript型定義方針

型定義は途中で迷い始めると急速に複雑化する。基本方針は次の2つだが、推奨は前者である。

**方針1: 最初から型セーフに進める（推奨）**

- メリット: 早い段階で整合性を確保でき、Cursor への指示精度も上がる
- メリット: CI/CD での型エラーを早期に検出しやすい
- デメリット: 初期設計に多少時間がかかる
- おすすめケース: 業務要件がある程度明確で、継続的に保守する画面

**方針2: 一旦 `any` 型で進める（非推奨）**

- メリット: 試作は速い
- デメリット: 後から型を付け直すコストが高い
- デメリット: Cursor が周辺コードとの整合を保ちにくくなる
- 注意点: 一時しのぎとして使うほど、最終的な修正範囲が広がりやすい

> 途中から場当たり的に型を付けると、かえって複雑化し、CI/CD エラーの原因になりやすい。

#### ⑥ モックデータを活用したUI再現

Figma の見た目を MUI で再現するときは、モックデータを積極的に使うとよい。空のフォームだけではなく、実際の入力値や長めの文字列、エラー表示状態、編集状態を持つデータで確認すると、UI の完成度が上がる。

#### ⑦ 段階的な改善とCursorとの対話

一度の指示で完成させようとせず、生成結果を見ながら Cursor と対話的に改善していくのが現実的である。余白、ラベル位置、バリデーション文言、レスポンシブ時の崩れなどを順に直していくと、最終品質が安定する。

### ステップ3 定着フェーズ

MUI + Cursor でうまくいった進め方は、個人のノウハウで閉じないように定着させるのが重要だ。具体的には次のような資産を残す。

- どのファイルを `@` で参照させると精度が上がるか
- どの粒度でコンポーネント分割すると修正しやすいか
- 型定義をどこまで先に固めるべきか
- レビューで何を重点確認するか

この流れをチームで共有すれば、単発の AI 活用ではなく、再現性のある開発手法として運用しやすくなる。

## まとめ

Figma 中心の開発で感じやすい壁に対して、MUI と Cursor の組み合わせはかなり有効だった。重要なのは、単に AI にコードを書かせることではなく、業務理解を整理し、関連ファイルを文脈として渡し、生成結果を必ずレビューすることだ。

そして、ここで紹介した Cursor の活用法は Cursor 固有の話に閉じない。文脈の渡し方、段階的な移行、レビュー前提の運用という考え方は、他の AI ツールにも十分応用できる。
